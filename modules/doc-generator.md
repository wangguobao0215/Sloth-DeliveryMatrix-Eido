# Doc Generator - 文档生成引擎

> 模块类型: 能力模块 (Capability Module)
> 调用方式: 被角色模块路由调用，或用户直接请求"生成文档""写报告""出材料"

## 1. 模块职责

本模块负责生成项目交付过程中的各类文档，支持多行业模板适配、数据驱动内容填充和自动化质量检查。

## 2. 模板选择逻辑

根据 `project.industry` 自动选择模板族：

```
模板选择流程:
1. 读取 project.industry
2. 匹配模板族:
   - industry == "saas" → templates/saas/
   - industry == "gov" → templates/gov/
   - industry == "finance" → templates/finance/
   - 其他或未指定 → templates/_base/
3. 在模板族内匹配文档类型
4. 如果模板族中无该文档类型，回退到 templates/_base/
```

**行业差异说明**：
| 行业 | 特殊要求 |
|------|---------|
| SaaS | 强调迭代发布节奏、版本号管理、SLA指标 |
| 政府 (gov) | 合规性要求高、档案格式规范、需要红头文件格式、签章页 |
| 金融 (finance) | 安全合规（等保）、审计追踪、数据脱敏要求 |
| 通用 (_base) | 标准软件交付文档格式 |

## 3. 文档类型表

| 文档类型 ID | 文档名称 | 使用场景 | 关键章节 | 典型页数 |
|------------|---------|---------|---------|---------|
| `weekly_report` | 项目周报 | PM每周汇报 | 进度概览、本周完成、风险与问题、下周计划 | 2-3页 |
| `brd` | 业务需求文档 | 需求阶段 | 业务背景、目标、范围、业务流程、非功能需求 | 10-20页 |
| `srs` | 软件需求规格说明 | 设计阶段 | 功能需求列表、接口说明、数据字典、约束条件 | 15-30页 |
| `test_report` | 测试报告 | 测试阶段 | 测试范围、用例统计、缺陷分析、结论与建议 | 5-10页 |
| `acceptance_report` | 验收报告 | 验收阶段 | 验收标准、测试结果、遗留问题、验收结论 | 8-15页 |
| `review_presentation` | 评审汇报材料 | 里程碑评审 | 项目概况、进展、风险、演示、下阶段计划 | 10-15页PPT |
| `director_weekly` | 管理层周报 | Director汇报 | 组合概览、健康度矩阵、风险TOP3、资源动态 | 3-5页 |
| `change_request` | 变更请求单 | 变更管理 | 变更描述、影响分析、审批意见 | 1-2页 |

## 4. 写作规则

所有文档生成必须遵循以下写作规则：

### 4.1 数据优先原则
- 每个论点必须有数据支撑，不使用"大约""估计""差不多"等模糊表述
- 如数据缺失，明确标注 `[数据待补充]` 而不是编造
- 百分比、日期、人数等关键数据必须精确

### 4.2 结论先行原则
- 每个章节的第一段必须是结论/总结
- 详细数据和分析放在结论之后
- 高管文档尤其重要：第一页必须能看到全貌

### 4.3 风险高亮原则
- 风险和问题使用醒目标记（红色/加粗/⚠️）
- 不要把风险埋在正文中间，使用独立章节或表格突出
- 每个风险必须附带建议的应对措施

### 4.4 行动导向原则
- 每个文档的结尾必须有"下一步行动"或"待办事项"
- 行动项必须明确：责任人 + 完成时限 + 具体内容
- 避免模糊的行动项如"持续关注""继续推进"

### 4.5 格式规范
- 日期统一使用 `YYYY-MM-DD` 格式
- 项目名称使用全称，首次出现后可用缩写（需标注）
- 表格优于段落，用表格呈现结构化信息
- 每页/每节控制在合理长度，避免信息过载

## 5. 自检清单

文档生成完成后，必须执行以下自检：

```
## 文档自检清单

### 内容完整性
- [ ] 所有章节标题已填充内容（无空章节）
- [ ] 所有 [待补充] 占位符已标注或已填充
- [ ] 项目名称、日期、版本号正确
- [ ] 关键数据已核实（进度百分比、缺陷数、人天数）

### 格式规范
- [ ] 日期格式统一 (YYYY-MM-DD)
- [ ] 表格对齐、无空行
- [ ] 标题层级正确（不跳级）
- [ ] 页眉页脚信息正确（如适用）

### 行动项
- [ ] 每个行动项有责任人
- [ ] 每个行动项有完成时限
- [ ] 行动项描述具体可执行

### 风险与问题
- [ ] 风险已按严重程度排序
- [ ] 每个风险有应对措施
- [ ] 已关闭的风险标注"已关闭"

### 行业合规（按需）
- [ ] [gov] 档案格式符合要求
- [ ] [gov] 涉密内容已标注密级
- [ ] [finance] 敏感数据已脱敏
- [ ] [finance] 审计追踪信息完整
```

## 6. 脚本集成

### 6.1 DOCX 生成

当用户需要导出为 Word 文档时，调用脚本：

```
脚本: scripts/generate_docx.py
参数:
  --template: 模板路径 (templates/{industry}/{doc_type}.docx)
  --data: JSON 格式的数据文件路径
  --output: 输出文件路径
  --logo: 可选，公司Logo路径

使用示例:
python scripts/generate_docx.py \
  --template templates/gov/acceptance_report.docx \
  --data /tmp/project_data.json \
  --output /tmp/验收报告_智慧园区_2024-01-15.docx
```

### 6.2 数据准备

在调用脚本前，需要将项目数据整理为 JSON 格式：

```json
{
  "project_name": "智慧园区一期",
  "project_code": "SP-2024-001",
  "client_name": "XX市住建局",
  "report_date": "2024-01-15",
  "author": "张三",
  "milestones": [...],
  "risks": [...],
  "action_items": [...]
}
```

## 7. Zero-State 降级处理

当项目状态文件中数据不足时，文档生成采用渐进降级策略：

| 缺失数据 | 降级行为 |
|---------|---------|
| milestones 为空 | 里程碑状态表替换为："[尚未录入里程碑，建议先设置关键节点]"，同时附加推荐操作 |
| risks 为空 | 风险章节替换为："[暂无已登记风险。建议执行一次风险扫描：发送"扫描风险"]" |
| team.members 为空 | 资源相关字段标注"[团队成员未录入]"，跳过工时统计 |
| weekly_events 为空 | 本周进展章节提示："[本周暂无录入事项。可以告诉我本周做了什么，或发送会议纪要/聊天记录]" |
| 全部为空（仅有项目名） | 生成"骨架版"文档：只有章节标题和占位提示，引导用户逐步补充数据 |

降级输出统一标注 `[基于有限输入，建议补充后重跑]`。

## 8. 周报快速生成流程

当用户说"帮我写周报"时的标准处理流程：

1. **检查数据源**: 确认项目状态数据是否可用，缺失则要求用户补充
2. **选择模板**: 根据角色选择 `weekly_report`（PM）或 `director_weekly`（Director）
3. **填充内容**: 基于项目状态自动填充进度、风险、里程碑
4. **补充分析**: 添加趋势分析（与上周对比）和建议
5. **自检**: 执行自检清单
6. **输出**: 先输出 Markdown 预览，用户确认后可导出 DOCX
