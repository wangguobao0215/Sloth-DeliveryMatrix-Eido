# Resource Coordinator - 资源协调能力模块

> 模块类型: 能力模块 (Capability Module)
> 调用方式: 被角色模块路由调用，或用户直接请求"资源分配""人力情况""谁有空""调配人员"

## 1. 模块职责

本模块负责对项目交付团队的人力资源进行分析、优化和协调建议。覆盖工作量分析、超载/闲置检测、技能匹配和资源调配建议。

## 2. 工作量分析

### 2.1 人员分配率计算

```
分配率 = Σ(各项目分配比例) × 100%

示例:
  张三: 项目A(60%) + 项目B(40%) = 100% (满载)
  李四: 项目A(80%) + 项目C(60%) = 140% (超载 ⚠️)
  王五: 项目B(30%) = 30% (低负载)
```

### 2.2 工作量分析表

输出标准格式：

```
## 团队资源分配概览

### 人员分配矩阵
| 姓名 | 角色 | 项目A | 项目B | 项目C | 总分配率 | 状态 |
|------|------|-------|-------|-------|---------|------|
| 张三 | 后端开发 | 60% | 40% | - | 100% | ✅ 满载 |
| 李四 | 前端开发 | 80% | - | 60% | 140% | 🔴 超载 |
| 王五 | 测试 | - | 30% | - | 30% | 🟡 低负载 |
| 赵六 | 后端开发 | - | - | 100% | 100% | ✅ 满载 |

### 汇总统计
- 团队总人数: 4
- 平均分配率: 92.5%
- 超载人数: 1 (占 25%)
- 低负载人数: 1 (占 25%)
```

### 2.3 分配率阈值

| 分配率范围 | 状态 | 含义 | 建议 |
|-----------|------|------|------|
| 0% | 空闲 | 未分配任何项目 | 考虑分配到缺人项目 |
| 1%-50% | 低负载 | 有较多空余产能 | 可承接新任务或支援其他项目 |
| 51%-80% | 正常 | 有一定缓冲 | 合理范围 |
| 81%-100% | 满载 | 无额外产能 | 避免加派任务 |
| 101%-120% | 超载 | 轻度超载 | 需关注，短期可接受（≤2周） |
| >120% | 严重超载 | 不可持续 | 必须立即调整 |

## 3. 超载/闲置检测

### 3.1 超载检测

当检测到人员超载时，自动生成告警：

```
## ⚠️ 资源超载告警

### 超载人员
| 姓名 | 总分配率 | 涉及项目 | 持续时间 | 风险 |
|------|---------|---------|---------|------|
| 李四 | 140% | A(80%)+C(60%) | 已持续3周 | 质量下降、倦怠 |

### 建议处理方案
1. **方案A - 降低分配**: 将李四在项目C的分配从60%降至40%，差额由新人补充
2. **方案B - 任务转移**: 将项目A中非核心模块转交其他成员
3. **方案C - 临时增援**: 从外部/其他项目借调1人，为期2周
```

### 3.2 闲置检测

当检测到人员闲置时，输出可分配建议：

```
## 💡 可用资源

| 姓名 | 当前分配率 | 可用产能 | 技能 | 建议分配 |
|------|-----------|---------|------|---------|
| 王五 | 30% | 70% | 测试、自动化 | 建议支援项目A的测试阶段 |
```

## 4. 技能匹配分析

### 4.1 技能矩阵

维护团队技能矩阵，用于资源调配时的技能匹配：

```
## 团队技能矩阵

| 姓名 | Java | Python | Vue | React | 测试 | 数据库 | DevOps | 行业经验 |
|------|------|--------|-----|-------|------|--------|--------|---------|
| 张三 | ★★★ | ★★ | - | - | ★ | ★★★ | ★ | 政府 |
| 李四 | - | ★ | ★★★ | ★★ | - | ★ | - | SaaS |
| 王五 | ★ | ★★ | - | - | ★★★ | ★ | ★★ | 金融 |

★ = 基础  ★★ = 熟练  ★★★ = 精通
```

### 4.2 技能匹配度计算

当需要为项目分配人员时，计算匹配度：

```
匹配度 = Σ(项目需求技能权重 × 候选人技能等级) / Σ(项目需求技能权重 × 最高等级)

示例: 项目A需要 Java(权重3) + 数据库(权重2)
  - 张三: (3×3 + 2×3) / (3×3 + 2×3) = 15/15 = 100% 完美匹配
  - 王五: (3×1 + 2×1) / (3×3 + 2×3) = 5/15 = 33% 匹配度低
```

### 4.3 技能缺口分析

```
## 技能缺口分析: [项目名称]

| 所需技能 | 需求等级 | 团队最高等级 | 差距 | 风险 | 建议 |
|---------|---------|------------|------|------|------|
| Java | ★★★ | ★★★ | 无 | 低 | - |
| K8s | ★★ | ★ | 1级 | 中 | 安排培训或外部支援 |
| 信创适配 | ★★ | - | 2级 | 高 | 必须外部引入有经验人员 |
```

## 5. 两种运行模式

### 5.1 单项目模式 (single_project)

分析单个项目的团队资源情况：

输出内容：
1. 项目团队成员列表及分配率
2. 超载/闲置人员标注
3. 技能覆盖度分析
4. 关键岗位冗余度（是否有单点故障——某个关键模块只有一人负责）
5. 资源优化建议

**单点故障检测**:
```
如果某个关键模块（核心业务、核心接口、部署运维）仅有1人负责:
  → 标注 "⚠️ 单点风险: [模块名] 仅 [姓名] 一人负责"
  → 建议: 安排备份人员或知识转移
```

### 5.2 跨项目模式 (cross_project)

面向 Director 的全局资源视图：

```
## 资源全景

### 1. 资源池总览
- 总人数: 12
- 平均分配率: 87%
- 超载人数: 3 (25%)
- 可用产能: 约 2.5 人力当量

### 2. 各项目资源状态
| 项目 | 需求人力 | 实际人力 | 缺口 | 紧缺技能 |
|------|---------|---------|------|---------|
| 智慧园区 | 5 | 3.5 | -1.5 | Java后端 |
| CRM二期 | 3 | 3 | 0 | - |
| 数据中台 | 4 | 4.2 | +0.2 | - |

### 3. 资源调配建议
基于项目优先级和进度状态，建议:
1. CRM二期即将进入维护期，可释放1名前端→支援智慧园区
2. 数据中台略有富余，可释放0.2人力当量
```
