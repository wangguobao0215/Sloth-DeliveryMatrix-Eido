# Requirement Analyzer - 需求分析能力模块

> 模块类型: 能力模块 (Capability Module)
> 调用方式: 用户说"分析需求""梳理需求""客户要这些功能"或粘贴原始需求文本时触发

## 1. 模块职责

本模块负责将客户的原始、零散、模糊甚至矛盾的需求输入，通过结构化分析转化为可交付的需求文档（BRD/SRS）。

## 2. 输入处理

### 2.1 典型输入类型

| 输入类型 | 特征 | 处理策略 |
|---------|------|---------|
| 客户会议口述 | 零散、跳跃、夹杂业务背景 | 先提取需求条目，再追问确认 |
| 邮件/微信内容 | 非结构化文本，可能多轮对话 | 合并同一需求的描述，去重 |
| 竞品截图/功能列表 | "我要做一个跟XX一样的" | 功能拆解 + 差异化分析 |
| 已有文档（老系统） | 可能过时、遗漏或包含已废弃功能 | 标注需确认项，区分沿用/废弃/新增 |
| 模糊表述 | "系统要好用""要快""要安全" | 转化为可量化指标（如响应时间<2秒）|

### 2.2 输入预处理步骤

1. **原文保留**: 保存客户原始表述，用于后续追溯
2. **需求条目拆分**: 将大段文字拆分为独立需求条目（一个条目 = 一个可测试的功能点）
3. **去重合并**: 相同或相似需求合并，标注来源
4. **分类标签**: 为每个需求添加分类（功能需求/非功能需求/业务规则/接口需求）
5. **模糊标注**: 对模糊/不明确的需求标注"需确认"

## 3. 需求分解流程

### 3.1 四层分解模型

```
客户原始输入
    ↓
第一层: 业务模块 (Business Module)
    例: 用户管理、订单管理、报表分析
    ↓
第二层: 功能点 (Feature)
    例: 用户注册、用户登录、密码重置
    ↓
第三层: 用户故事 (User Story)
    例: 作为普通用户，我想要通过手机号注册账号，以便使用系统功能
    ↓
第四层: 验收标准 (Acceptance Criteria)
    例: 
    - 输入11位手机号，发送验证码
    - 验证码60秒内有效
    - 密码需 ≥8位，含大小写和数字
```

### 3.2 分解输出格式

```
## 需求分解: [项目/模块名称]

### 模块 1: 用户管理

#### F01: 用户注册
**优先级**: 高
**用户故事**: 作为[角色]，我想要[动作]，以便[价值]

**验收标准**:
- AC01: [标准描述] → 可测试: [是/需细化]
- AC02: [标准描述] → 可测试: [是/需细化]

**业务规则**:
- BR01: [规则描述]

**非功能要求**:
- 并发: [描述或待确认]
- 响应时间: [描述或待确认]

**关联接口**: [如有]
**备注**: [客户原始表述/特殊要求]
```

## 4. 矛盾检测

### 4.1 常见矛盾类型

| 矛盾类型 | 描述 | 识别方法 | 示例 |
|---------|------|---------|------|
| 功能冲突 | 两个需求逻辑上互斥 | 语义分析 | "所有数据实时同步" vs "支持离线使用" |
| 范围冲突 | 需求范围与预算/工期不匹配 | 工作量估算对比 | 50个功能点 vs 2个月工期2人团队 |
| 角色冲突 | 不同干系人的需求相互矛盾 | 来源追溯 | 业务部要灵活 vs IT部要标准化 |
| 优先级冲突 | 多个需求都标记为"最高优先级" | 统计分析 | 10个需求全标P0 |
| 约束冲突 | 技术约束与业务需求不兼容 | 技术可行性检查 | "用微信小程序实现AR功能" |

### 4.2 矛盾输出格式

```
## ⚠️ 需求矛盾检测

### 矛盾 1: [简述]
- **需求A**: [描述] (来源: [谁/何时提出])
- **需求B**: [描述] (来源: [谁/何时提出])
- **矛盾性质**: [功能冲突/范围冲突/...]
- **影响**: [如不解决会导致什么]
- **建议**: [解决方案建议]
```

## 5. GAP 分析（标准产品 vs 客户需求）

### 5.1 适用场景

当项目基于已有 SaaS 产品进行定制化交付时，执行 GAP 分析。

### 5.2 分析流程

```
1. 梳理客户需求完整清单
2. 逐条对照标准产品功能
3. 标注差距类型和解决策略
4. 估算定制开发工作量
5. 输出GAP分析报告
```

### 5.3 差距类型与解决策略

| 差距类型 | 定义 | 解决策略 | 典型工作量 |
|---------|------|---------|-----------|
| 无差距 | 标准产品完全满足 | 直接使用 | 0 |
| 配置型 | 通过配置/参数可满足 | 系统配置 | 0.5-2人天 |
| 轻量定制 | 需少量前端/报表定制 | 低代码/脚本 | 2-5人天 |
| 标准开发 | 需要新功能模块开发 | 标准开发流程 | 5-20人天 |
| 架构型 | 需要架构层面调整 | 深入评估 | 20+人天 |
| 不可行 | 技术或架构上无法实现 | 替代方案协商 | N/A |

### 5.4 GAP 分析报告模板

```
## GAP 分析报告: [项目名称]

### 基本信息
- 标准产品: [产品名 + 版本]
- 客户: [客户名]
- 分析日期: [日期]
- 需求条目总数: [N]

### GAP 统计
| 差距类型 | 数量 | 占比 | 预估总工作量 |
|---------|------|------|------------|
| 无差距 | X | X% | 0 |
| 配置型 | X | X% | X人天 |
| 轻量定制 | X | X% | X人天 |
| 标准开发 | X | X% | X人天 |
| 架构型 | X | X% | 待评估 |
| 不可行 | X | X% | N/A |

**定制化率**: [有差距的需求数/总需求数 × 100%]
**预估定制开发总工作量**: [X 人天 (±30%)]

### GAP 明细
[详细表格，每条需求一行]

### 风险提示
- 定制化率 > 40%: 建议评估是否适合使用该标准产品
- 架构型差距 > 2处: 建议进行技术可行性评审
- 不可行项 > 0: 必须与客户协商替代方案
```

## 6. 需求文档输出

### 6.1 BRD（业务需求文档）结构

当用户需要输出 BRD 时，使用以下结构：

```
1. 文档信息（版本、日期、作者、审批人）
2. 项目背景与目标
3. 业务范围
4. 干系人分析
5. 业务流程描述（现状 + 期望）
6. 功能需求概览（模块级别）
7. 非功能需求
8. 约束与假设
9. 风险与依赖
10. 验收标准
11. 附录
```

### 6.2 SRS（软件需求规格说明）结构

```
1. 文档信息
2. 系统概述
3. 功能需求（按模块组织，含用户故事和验收标准）
4. 接口需求（内部接口 + 外部接口）
5. 数据字典
6. 非功能需求（性能、安全、可用性、兼容性）
7. 约束条件
8. 需求追踪矩阵（需求ID ↔ 设计/测试用例映射）
```

## 7. 需求质量评估

完成需求分析后，对需求整体质量进行评估：

| 评估维度 | 评分标准 | 权重 |
|---------|---------|------|
| 完整性 | 是否覆盖所有业务场景 | 25% |
| 一致性 | 需求之间是否无矛盾 | 20% |
| 可测试性 | 每个需求是否有明确验收标准 | 20% |
| 可追溯性 | 每个需求是否有来源记录 | 15% |
| 明确性 | 是否无歧义、无模糊表述 | 20% |

```
需求质量总分 = Σ(各维度评分 × 权重)
  ≥ 80分: 可进入设计阶段
  60-79分: 需补充完善后方可进入设计
  < 60分: 需重新梳理
```
