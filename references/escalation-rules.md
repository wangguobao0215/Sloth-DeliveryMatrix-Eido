# 升级规则 (Escalation Rules)

> 定义项目交付过程中风险与问题的升级阈值、路径、报告格式和降级条件。

---

## 一、升级阈值

以下任一条件触发时，必须启动升级流程：

| 类别 | 指标 | 阈值 | 升级级别 |
|------|------|------|---------|
| 进度风险 | 计划偏差（SPI） | SPI < 0.85 或延期 > 15% | L1 → 项目总监 |
| 预算风险 | 成本偏差（CPI） | CPI < 0.90 或超支 > 10% | L1 → 项目总监 |
| 质量风险 | 严重缺陷未关闭 | P0/P1 缺陷 > 3 天未解决 | L1 → 技术总监 |
| 质量风险 | 缺陷密度 | 缺陷密度 > 5 个/千行代码 | L1 → 技术总监 |
| 资源风险 | 关键人员离场 | 核心角色（架构师/TL）离场 | L2 → VP |
| 范围风险 | 需求变更频率 | 单迭代变更 > 计划范围的 20% | L1 → 项目总监 |
| 外部风险 | 甲方阻塞 | 甲方依赖项 > 5 个工作日未响应 | L1 → 项目总监 |
| 合同风险 | 里程碑逾期 | 合同里程碑延期 > 5 个工作日 | L2 → VP |

---

## 二、升级路径

```
项目经理 (PM)
    │
    ▼  L1 升级
项目总监 / 技术总监 (Director)
    │
    ▼  L2 升级（L1 升级后 48 小时内未解决，或触发 L2 阈值）
副总裁 / 事业部负责人 (VP)
    │
    ▼  L3 升级（极端情况：合同违约风险、重大安全事故）
CEO / 管理委员会
```

**升级时效要求**：
- L1 升级：识别后 **4 小时内** 完成上报。
- L2 升级：L1 升级后 **48 小时** 未有效缓解，自动升至 L2。
- 每次升级必须附带升级报告。

## 三、升级报告模板

```markdown
## 升级报告

**项目名称**：{project_name}
**报告日期**：{date}
**升级级别**：L1 / L2 / L3
**报告人**：{reporter}

### 1. 问题概述
- 一句话描述当前问题及其影响

### 2. 关键数据
- 触发指标：{metric} = {value}（阈值：{threshold}）
- 影响范围：{scope}
- 已持续时间：{duration}

### 3. 已采取措施
- {action_1}：{result}
- {action_2}：{result}

### 4. 需要的支持
- {support_request_1}
- {support_request_2}

### 5. 建议方案
- 方案 A：{description}（预期效果/风险/成本）
- 方案 B：{description}（预期效果/风险/成本）

### 6. 决策时限
- 需在 {deadline} 前做出决策，否则 {consequence}
```

---

## 四、降级条件

满足以下全部条件时，可执行降级：

1. 触发升级的指标已恢复到阈值以内，且稳定 **5 个工作日** 以上。
2. 根因已消除或有效缓解措施已落地。
3. 相关干系人确认同意降级。
4. 降级记录存档，包含问题总结和改进措施。

## 五、中国企业 IM 工具中的升级实践

### 钉钉 (DingTalk)
- 使用 **DING 消息** 确保必达，升级通知用 DING 而非普通消息。
- 建立专项升级群，包含决策层，仅在触发升级时激活。
- 利用钉钉审批流创建正式的升级审批单，留存记录。

### 飞书 (Feishu/Lark)
- 使用 **加急消息** 功能标记升级通知。
- 在飞书文档中维护升级跟踪表，利用 @mention 通知相关人。
- 通过飞书多维表格自动化升级状态跟踪。

### 企业微信 / 云之家
- 升级消息使用 **应用消息推送** 确保触达，避免淹没在群聊中。
- 重要升级同步发送邮件作为正式记录。
- 利用审批流程模块创建升级工单，确保流程合规可追溯。

### 通用建议
- 升级消息格式统一，开头标注 **【升级通知-L{级别}】**，便于快速识别。
- 所有 IM 升级通知同步归档到项目管理系统（如禅道、Jira），避免信息散落。
